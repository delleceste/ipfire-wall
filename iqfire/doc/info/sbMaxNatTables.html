Maximum number of NAT tables in the kernel

When a packet is subjected to Network address (and/or port) translation, an associated NAT table is created inside the ipfire-wall kernel, to track the connection and apply the NAT to all the other packets pertaining to the same connection flow.

It is possible to tune the maximum number of NAT table entries, so that their numerousness does not grow over that limit. This was done thinking about a possible denial of service attack, in which an attacker could fill up the kernel memory with useless NAT tables.

Tune this value to the estimated amount of NAT traffic you expect for your linux NAT/firewall, keeping in mind that when the limit is reached NAT stops being applied until the timer of some other entry in the NAT table list expires and frees the associated memory.

Lifetime of NAT entries

The lifetime of NAT entries depends on the state of the connection and so it is managed internally by the kernel. On the command line interface ipfire, it is possible to list the kernel NAT tables by pressing the keys F4 to view the source NAT entries and F6 to observe the destination NAT entries. At the moment, this is considered an advanced feature and so it is not included in the iqfire graphical user interface.
